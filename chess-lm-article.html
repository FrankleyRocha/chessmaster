Introdução

Xadrez como problema de linguagem

Uma partida de xadrez em notação PGN é, na essência, uma sequência de símbolos. 1. e4 e5 2. Nf3 — cada movimento é um token, cada partida é uma sentença. A pergunta que guia este projeto é direta: dado tudo que aconteceu no tabuleiro até agora, qual é o próximo caractere provável?

Essa reformulação transforma o problema de xadrez num problema de modelagem de sequências — exatamente o que transformers decoder-only fazem melhor. Não há motor de busca, não há árvore minimax. Apenas padrões aprendidos de milhares de partidas de jogadores de elite.

Dataset · Fine-tuning

Os seis estilos que definem o modelo

O fine-tuning é feito sobre partidas de seis jogadores com estilos radicalmente distintos. O modelo aprende a transitar entre precisão cirúrgica, caos calculado e elegância posicional — tudo na mesma base de pesos.

Tokenização

Vocabulário de ~50 caracteres

A tokenização é feita ao nível de caractere. Isso resolve nativamente todos os edge cases da notação algébrica — xeques (+), mates (#), promoções (e8=Q), roque longo (O-O-O) — sem nenhum tratamento especial.

Por que caractere e não subword? Com vocabulário de ~50 símbolos, tokenização por caractere é trivialmente eficiente. BPE ou WordPiece não trazem vantagem aqui — só complexidade desnecessária.
Arquitetura

GPT-like, decoder-only

A arquitetura segue o padrão GPT: transformer decoder com atenção causal. O modelo prevê o próximo caractere dado todo o contexto anterior — autoressivamente, da abertura até o último lance.

Partidas têm em média 40–80 movimentos, o que equivale a ~300–600 caracteres. Contexto curto, modelo pequeno é suficiente.

# Hiperparâmetros sugeridos

vocab_size = 64 # ~50 chars + padding + tokens especiais
block_size = 512 # contexto máximo em caracteres
n_embd = 256 # dimensão do embedding
n_head = 8 # cabeças de atenção
n_layer = 6 # blocos transformer
dropout = 0.1 # regularização

# Estimativa: ~10M parâmetros
Estratégia de Treino

Pré-treino + Fine-tuning

O treinamento acontece em duas fases. Na primeira, o modelo aprende a "falar xadrez" — sintaxe válida, padrões gerais de abertura, meio-jogo e endgame. Na segunda, ele especializa no estilo dos seis jogadores escolhidos.

Fase 1
Pré-treino — Lichess Database
Filtro aplicado: Elo ≥ 2500 para ambos os jogadores + apenas vitórias (sem empates). O modelo aprende movimentos de alto nível — cada sequência no dataset veio de um vencedor forte. Volume: centenas de milhares de partidas disponíveis.
Fase 2
Fine-tuning — Os Seis Gênios
Partidas de Fischer, Kasparov, Tal, Capablanca, Pragg e Magnus — misturadas sem distinção de jogador. O modelo especializa o conhecimento geral de xadrez nos padrões idiossincráticos desses seis estilos. ~10.000 partidas no total.
Por que essa ordem importa? Sem o pré-treino, ~10k partidas dificilmente são suficientes para o modelo aprender movimentos válidos de forma consistente. O pré-treino resolve a gramática do xadrez; o fine-tune resolve o estilo.
Implementação

Pipeline do projeto

01
Download e filtragem do Lichess Script Python que parseia os dumps mensais em .pgn.zst, filtra por Elo ≥ 2500 e resultado 1-0 ou 0-1, e salva as partidas em texto plano.
02
Download das partidas dos jogadores Coleta dos arquivos .pgn do PGN Mentor para os seis jogadores. Limpeza e normalização do formato.
03
Tokenizador por caractere Mapeamento bidirecional char → int e int → char. Construção do vocabulário a partir do corpus completo.
04
Implementação do modelo GPT-like em PyTorch seguindo a arquitetura definida. Atenção causal, positional encoding, blocos transformer.
05
Pré-treino no dataset Lichess Treino no corpus filtrado de partidas 2500+. Objetivo: cross-entropy loss no próximo caractere.
06
Fine-tuning nos seis jogadores Continuação do treino com learning rate menor nas partidas dos gênios. Monitoramento de overfitting.
07
Avaliação Percentual de movimentos válidos gerados, perplexidade no conjunto de teste, comparação qualitativa de aberturas.
Formato

Input e output do modelo

O modelo recebe a partida até o momento como string de caracteres e prevê o próximo. Na inferência, gera autoregressivamente até o final da partida.

# Input: partida até o momento
"1. e4 e5 2. Nf3 Nc6 3. Bb5 a6 4. Ba4 Nf3"

# O modelo prevê o próximo caractere:
→ " "   (espaço)
→ "N"
→ "c"
→ "6"

# Output: partida estendida com o próximo movimento
"1. e4 e5 2. Nf3 Nc6 3. Bb5 a6 4. Ba4 Nf3 Nc6"

Cada chamada de inferência estende a sequência em um caractere. Para gerar um movimento completo, o processo roda até encontrar o próximo espaço separador.
